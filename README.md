
# Report Export Service

## Описание

Сервис предоставляет API для анализа текстовых файлов и построения частотной статистики по словоформам.

Эндпоинт:
`POST /public/report/export`

Принимает текстовый файл и возвращает Excel-файл (xlsx), содержащий:

* словоформу (нормальную форму слова)
* общее количество вхождений в документе
* количество вхождений в каждой строке (в формате строки, например: `0,11,32,0,0,3`)

Поддерживаются большие файлы: обработка выполняется построчно без загрузки всего файла в память.

---

## Ограничения и валидация

Сервис накладывает следующие ограничения:

* принимаются только текстовые файлы в кодировке **UTF-8**
* пустые файлы отклоняются
* установлен лимит на размер файла (по умолчанию — **50 MB**)
* при превышении лимита возвращается `413 Payload Too Large`

## Запуск проекта

1. Клонировать репозиторий

```
git clone https://github.com/annshev1avis/test_kaspersky_backend
cd test_kaspersky_backend
```

2. Создать виртуальное окружение

```
Windows:
py -m venv venv

Linux/MacOS:
python3 -m venv venv
```

3. Активировать виртуальное окружение

```
Windows:
venv\Scripts\activate

Linux / macOS:
source venv/bin/activate
```

```
pip install -r requirements.txt
```

5. Запустить сервер

```
uvicorn app.main:app --reload
```

6. Открыть Swagger UI

```

```

## Стек и принятые решения

* **FastAPI** — для реализации HTTP API
* **pymorphy3** — для лемматизации (нормализации словоформ)
* **openpyxl** — для генерации Excel-файлов

### Почему pymorphy3

Библиотека выбрана как лёгкое и эффективное решение для русского языка:

* не требует загрузки тяжёлых ML-моделей
* быстро работает “из коробки”
* обеспечивает достаточную точность для задачи частотного анализа

Альтернативы (например, spaCy) были бы избыточны для данного кейса.

## Обработка нагрузки и устойчивость

Так как анализ файла является ресурсоёмкой операцией (CPU-bound), в сервисе реализованы механизмы защиты от перегрузки:

### 1. Ограничение параллельной обработки

Используется `asyncio.Semaphore`, ограничивающий количество одновременно выполняемых задач анализа.

* если лимит занят, новые запросы не ставятся в очередь
* сервер сразу возвращает `503 Service Unavailable`

Это предотвращает ситуацию, когда несколько больших файлов полностью блокируют приложение.

---

### 2. Вынесение тяжёлой обработки из event loop

Синхронная обработка файла (`create_report`) выполняется через:

```python
run_in_threadpool(...)
```

Это позволяет:

* не блокировать event loop FastAPI
* сохранять отзывчивость API при длительной обработке

Важно: данный подход **не ускоряет вычисления**, но повышает устойчивость сервиса.

---

### 3. Работа с временными файлами

Excel-файл создаётся как временный и:

* отправляется пользователю
* автоматически удаляется после ответа

Это предотвращает накопление файлов на диске.

---

### 4. Масштабирование через воркеры

Сервис может быть запущен с несколькими воркерами:

```bash
uvicorn app.main:app --workers 4
```

Важно:

* ограничение через semaphore действует **в рамках одного процесса**
* итоговое количество одновременно выполняемых задач масштабируется пропорционально числу воркеров
